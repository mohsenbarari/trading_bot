## Accepted Findings

### Staging trusted proxy configuration gap

Claude's concern about staging proxy attribution was valid. The trusted-proxy
parser in `core/request_logging.py` is correct, but staging env generation did
not set `TRUSTED_PROXY_CIDRS`. In a host-nginx-to-Docker-container topology that
can collapse request attribution to the Docker gateway address.

Actions taken:

- `scripts/deploy_staging.sh` now defines `STAGING_TRUSTED_PROXY_CIDRS` with a
  default of loopback plus the Docker bridge private range:
  `127.0.0.1/32,::1/128,172.16.0.0/12`.
- New `.env.staging` files generated by `scripts/deploy_staging.sh` include
  `TRUSTED_PROXY_CIDRS`.
- Existing `.env.staging` files are patched by `ensure_runtime_env_values()`.
- `deploy/staging/env.staging.example` documents the required value.
- `deploy/staging/README.md` explains why this value is needed and when it may
  be overridden.
- `tests/test_deploy_surface_smoke.py` now asserts that the staging script and
  example env preserve this proxy configuration.

### CORS empty-environment fail-open default

Claude's minor CORS concern was valid. Production normally sets
`ENVIRONMENT=production`, but treating an empty environment as local/dev was
unnecessary and fail-open.

Actions taken:

- `core/deployment_surface.py` no longer treats an empty environment as local.
- `tests/test_deployment_surface.py` now has explicit coverage proving empty
  environment does not add localhost origins.
- The existing localhost-inclusion test was made explicit with
  `environment="staging"`.

## Rejected Or Qualified Findings

### OTP per-IP throttle raw-peer claim

Claude reported that OTP per-IP lock keys still use the raw TCP peer. That is
not true in the current checkout. `api/routers/auth.py` already routes
`_request_client_host()` through `client_ip_from_request(raw_request)`, which is
the trusted-proxy-aware parser added during VF7 remediation.

Current behavior:

- The OTP per-subject lock remains the primary brute-force protection.
- The OTP per-IP key uses trusted-proxy-aware client attribution when
  `TRUSTED_PROXY_CIDRS` is correctly configured.
- The confirmed remaining issue was configuration coverage in staging, not the
  auth helper implementation itself.

No auth-code change was made for this item in this pass.

### Production data hygiene guard execution status

Claude correctly noted that VF9 is detective and awaits routine production
release execution. That does not mean VF9 is incomplete from a code standpoint.
The guard is already read-only, container-safe, exposed through Make targets,
and wired into `production_deploy_online.sh` healthcheck.

Current status:

- Implementation is complete.
- First routine production execution remains an operational release step.
- No cleanup or data mutation is performed by the guard.

## Deferred Items

### Chat file authorization performance

The chat file authorization path is correct for object-level access, but the
message reference check still relies on matching the file id inside message
content. This is acceptable for the current release security posture, but it is
a performance debt for large message tables.

Recommended future work:

- Add an indexed message-attachment reference table or equivalent normalized
  lookup.
- Keep the current URL contract stable for the frontend.
- Preserve group/channel/direct membership semantics.

### GitHub provider-managed PR refs

Reachable branch and tag history has been rewritten for the leaked dev key, and
runtime keys were rotated. GitHub-managed `refs/pull/*` cleanup remains a
provider-side task if full provider-level erasure is required.
