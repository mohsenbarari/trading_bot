# پروتکل بازبینی خارجی WebApp UI/UX Redesign V2

تاریخ: ۲۰۲۶-۰۷-۱۸

وضعیت: الزام مالک محصول

## هدف

در پایان checkpointهایی که از نظر ریسک، اندازه یا اثر بین‌صفحه‌ای معنادار هستند، چهار دستیار مستقل تغییرات را با دامنه و شواهد یکسان اما متناسب با سطح دسترسی خود بررسی کنند. تصمیم نهایی همچنان بر عهده مجری اصلی و مالک محصول است.

## دستیارها و مرز دسترسی

| دستیار | دسترسی مبنا | قرارداد تحویل |
| --- | --- | --- |
| ChatGPT Ultra | GitHub و فایل‌های ارسالی مالک محصول | commit و branch پوش‌شده به‌علاوه ZIP شواهد خارج از Git |
| ChatGPT Pro | GitHub و فایل‌های ارسالی مالک محصول | commit و branch پوش‌شده به‌علاوه ZIP شواهد خارج از Git |
| Gemini | checkout و فایل‌سیستم کامل سرور | branch/commit دقیق و مسیر محلی شواهد خارج از Git |
| Claude | checkout و فایل‌سیستم کامل سرور | branch/commit دقیق و مسیر محلی شواهد خارج از Git |

## زمان ایجاد review gate

مجری اصلی در پایان یک یا چند checkpoint و بر اساس این نشانه‌ها review gate می‌سازد:

- تغییر در قرارداد مشترک طراحی، navigation، authentication، authorization یا state management؛
- تغییر چند صفحه یا component مشترک؛
- وجود ریسک دسترس‌پذیری، امنیت، از‌دست‌رفتن داده کاربر یا regression موبایل؛
- نیاز به قضاوت بصری یا UX قبل از گسترش الگو به صفحات بعدی؛
- حجم بالای کد یا شواهد تست که یک بازبینی مستقل را توجیه می‌کند.

تغییر مستندی کوچک، اصلاح تایپی یا artifact موقت کم‌ریسک می‌تواند تا gate بعدی تجمیع شود؛ دلیل تجمیع باید در handoff ثبت شود.

## محتوای الزامی هر handoff

- branch، commit و commit range دقیق؛
- هدف محصول، دامنه تغییر و non-goalها؛
- تصمیم‌های قبلی مالک محصول و قراردادهای غیرقابل‌تغییر؛
- فهرست فایل‌های اصلی و نقاط پرریسک؛
- دستورهای تست اجراشده، خروجی و محدودیت شواهد؛
- تصاویر، لاگ‌ها، full-matrix، گزارش‌های تولیدشده یا هر artifact خارج از Git؛
- پرسش‌های صریح برای correctness، regression، accessibility، UX و مرز بازار/پیام‌رسان؛
- قالب گزارش Markdown و مسیر/نام فایل خروجی مورد انتظار.

## قواعد ZIP برای ChatGPT Ultra و ChatGPT Pro

- بسته در یک پوشه مشخص زیر `tmp/` ساخته می‌شود و یک `README-FIRST.md` دارد.
- prompt مشترک و prompt اختصاصی هر دستیار جدا هستند.
- فقط شواهد لازم برای commit مورد بررسی قرار می‌گیرند؛ فایل‌های حساس، token، credential و داده شخصی وارد بسته نمی‌شوند.
- بسته دارای `SHA256SUMS` داخلی، checksum مستقل ZIP و نتیجه موفق بررسی ساختار ZIP است.
- هر artifact خارج از Git با منشأ، زمان تولید و دستور بازتولید توضیح داده می‌شود.

## قواعد prompt

- prompt به زبان انگلیسی، مستقل و دارای context کامل نوشته می‌شود.
- دستیار باید ادعاهای مجری را راستی‌آزمایی کند و صرفاً خلاصه ارائه ندهد.
- یافته‌ها باید severity، evidence، impact و concrete recommendation داشته باشند.
- دستیار باید موارد بررسی‌نشده یا غیرقابل‌دسترسی را صریح اعلام کند و نباید اجرای تست یا مشاهده سرور را جعل کند.
- هر دستیار گزارش خود را در یک فایل `.md` مستقل تحویل می‌دهد.
- نام فایل، عنوان گزارش و نقش اعلام‌شده داخل گزارش باید با prompt همان دستیار منطبق باشد. خروجی با نقش یا ساختار دستیار دیگر، حتی اگر یافته‌های مفیدی داشته باشد، جای گزارش مستقل مورد انتظار را در review quorum نمی‌گیرد.

## رسیدگی به یافته‌ها

هیچ یافته‌ای صرفاً به دلیل نام یا اعتبار دستیار پذیرفته یا رد نمی‌شود. برای هر مورد:

1. ادعا با source، test، artifact یا رفتار قابل‌بازتولید بررسی می‌شود؛
2. نتیجه یکی از `accepted`، `partially accepted`، `rejected` یا `deferred` است؛
3. دلیل و شواهد تصمیم ثبت می‌شود؛
4. اگر یافته رد یا فقط بخشی از آن پذیرفته شد، prompt دور بعد باید همان یافته و دلیل تصمیم را برای جلوگیری از تکرار بدون شواهد ذکر کند.

خلاصه dispositionها باید در یک register نسخه‌بندی‌شده کنار artifact همان checkpoint نگهداری شود. گزارش‌های خام و بسته‌های حجیم می‌توانند خارج از Git باقی بمانند، اما شناسه finding، دستیار، تصمیم، دلیل، شواهد و carry-forward آن‌ها نباید فقط در `tmp/` باقی بماند.

## مرز محافظت‌شده

بازبینی UI/UX اجازه بررسی یا پیشنهاد تغییر محتوای داخلی بازار و پیام‌رسان را ندارد. اثر غیرمستقیم component یا token مشترک بر این دو سطح باید به‌عنوان ریسک گزارش شود، نه اینکه بدون مجوز با تغییر داخلی آن‌ها رفع شود.
