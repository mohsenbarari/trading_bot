# Stage 0B-1 — ورود، دعوت، ثبت‌نام و بازیابی

تاریخ: ۲۰۲۶-۰۷-۱۸

وضعیت: یافته‌های دور دوم disposition و artifact طراحی اصلاح شد؛ در انتظار تأیید مالک محصول و review gate چهار‌نقشی روی SHA نهایی، design-only و بدون مجوز تغییر کد محصول

جهت مصوب: **مالی مدرن**

شاخه: `condidate/webapp-ui-ux-redesign-v2`

## ۱. هدف checkpoint

پیش از تعمیم زبان «مالی مدرن» به پوسته احراز‌شده، مشخص شود کاربر در حساس‌ترین مسیر پیش از ورود:

- همیشه بداند در کدام مرحله است؛
- بین ورود کاربر موجود، ثبت‌نام دعوت‌شده و بازیابی نشست سردرگم نشود؛
- داده واردشده را در خطا یا جابه‌جایی مرحله از دست ندهد؛
- روی موبایل و هنگام بازبودن کیبورد به اقدام اصلی دسترسی داشته باشد؛
- وضعیت انتظار، انقضا، رد یا موفقیت را به‌وضوح تشخیص دهد.

## ۲. قرارداد رفتاری ثابت و مرزهای پذیرش

این Stage رفتار محصول را تغییر نمی‌دهد و prototype باید قرارداد فعلی را حفظ کند. هرجا یک رفتار هنوز در runtime فعلی اثبات نشده و برای پیاده‌سازی آینده لازم است، در همین بخش صریحاً با عنوان «معیار پذیرش پیاده‌سازی» از قرارداد فعلی جدا می‌شود:

### دعوت‌نامه

- لینک `/i/:code` ابتدا وضعیت دعوت‌نامه را بررسی می‌کند.
- دعوت‌نامه pending می‌تواند مسیر تلگرام، وب یا هر دو را مطابق پاسخ سرور نشان دهد.
- خرابی تنظیمات ربات فقط مسیر تلگرام را حذف می‌کند و نباید مسیر وب معتبر را از کار بیندازد.
- دعوت‌نامه completed مستقیماً به ورود OTP با پیام «ثبت‌نام قبلاً تکمیل شده است» می‌رود.
- expired، invalid یا pending بدون token به‌صورت fail-closed نمایش داده می‌شود.
- سطح عمومی دعوت فقط داده مجاز و ماسک‌شده قرارداد سرور را نمایش می‌دهد؛ token خام، شماره کامل یا هویت اضافه در UI افشا نمی‌شود.

### ورود کاربر موجود

- شماره موبایل به درخواست OTP می‌رسد.
- کد می‌تواند ابتدا از تلگرام تحویل شود و همان کد در زمان اعلام‌شده به‌صورت خودکار با پیامک fallback شود؛ در این حالت resend دستی نمایش داده نمی‌شود.
- قرارداد legacy می‌تواند resend دستی پیامک را بعد از deadline مجاز کند.
- تلاش OTP فعال پس از refresh با شناسه opaque بازگردانی می‌شود و شماره موبایل در session storage ذخیره نمی‌شود.
- نتیجه OTP یکی از ورود موفق، نیاز به تکمیل ثبت‌نام یا نیاز به تأیید دستگاه اصلی است.
- نصب PWA یک اقدام ثانویه و وابسته به قابلیت مرورگر است و نباید مزاحم ورود باشد.
- کنترل OTP ورود از نظر معنایی یک input واحد با `one-time-code` باقی می‌ماند؛ تقسیم‌بندی بصری نباید آن را به چند input مستقل تبدیل یا تفاوت قرارداد ورود و OTP دقیقاً پنج‌رقمی ثبت‌نام را حذف کند.

### تأیید دستگاه و بازیابی نشست

- کاربر می‌تواند منتظر تأیید دستگاه اصلی بماند یا اعلام کند به دستگاه قبلی دسترسی ندارد.
- بازیابی ممکن است در انتظار بررسی مدیریت بماند، مدرک هویتی درخواست کند، یا به approved، rejected، expired و cancelled برسد.
- انتخاب گالری، دوربین و فایل هر سه حفظ می‌شوند؛ توضیح مدرک اختیاری است.
- ورود نهایی فقط با token تأییدشده انجام می‌شود.
- امکان شروع recovery تابع سیاست نقش و نشست است؛ نمایش affordance به معنی تضمین مجازبودن آن برای همه کاربران، به‌ویژه مدیران، نیست و خطای عدم صلاحیت باید در همان context قابل‌فهم باشد.
- متن وضعیت فقط به به‌روزرسانی خودکار و بی‌نیازی از refresh دستی متعهد است و cadence داخلی polling را به کاربر وعده نمی‌دهد.
- متن مدرک هویتی فقط می‌گوید فایل برای همین درخواست به بازبینان مجاز ارسال می‌شود؛ تا پیش از وجود سیاست مصوب، درباره انحصاری‌بودن استفاده، مدت نگهداری یا حذف قطعی ادعایی نمی‌شود.

### ثبت‌نام وب

- `REG-01` دقیقاً سه مرحله الزامی دارد: مرور دعوت‌نامه، OTP پنج‌رقمی ثبت‌نام و تکمیل اطلاعات نهایی.
- اتصال تلگرام فقط بعد از تکمیل ثبت‌نام و ایجاد نشست پیشنهاد می‌شود؛ از progress سه‌مرحله‌ای بیرون است، شماره مرحله نمی‌گیرد و رد آن کاربر را از ورود به محصول منع نمی‌کند.
- مسیر `REG-02` که با `registration_token` از ورود OTP آمده، با عنوانی مانند «تکمیل اطلاعات پس از ورود» مستقیماً اطلاعات نهایی را تکمیل می‌کند؛ OTP تکراری، progress ساختگی «۳ از ۳» و Back به OTP ثبت‌نام در آن مجاز نیست.
- login OTP و registration OTP می‌توانند presentation بخش‌بندی‌شده مشترک داشته باشند، اما هرکدام یک input معنایی واحد با semantics مستقل خود باقی می‌مانند؛ قرارداد WebOTP ورود نباید ناخواسته به OTP ثبت‌نام تعمیم داده شود.
- شماره موبایل در UI هر دو مسیر ثبت‌نام ماسک‌شده نمایش داده می‌شود. اینکه پاسخ فعلی `registration_token` می‌تواند شماره کامل و token دعوت را به client برساند، بدهی payload جداگانه است و برای ماسک‌کردن نمایش به تغییر endpoint در این Stage نیاز نیست.
- متن آدرس درباره تکمیل حساب و استفاده/نمایش بعدی آن مطابق قرارداد واقعی محصول صادق و محدود است و از عبارت مطلق «فقط برای تکمیل حساب» استفاده نمی‌کند.
- افشای consequential آدرس کنار همان فیلد باقی می‌ماند و افشای مدرک هویتی کنار context انتخاب/ارسال فایل قرار می‌گیرد؛ انتقال آن‌ها فقط به footer دور از اقدام مجاز نیست.
- نگهداری داده، Back و بازیابی flow پس از refresh یا خطای موقت، معیار پذیرش پیاده‌سازی آینده است؛ این checkpoint آن را به‌عنوان رفتار اثبات‌شده runtime فعلی معرفی نمی‌کند.

### تنظیم رمز عبور

- این مسیر فقط برای کاربر احراز‌شده است.
- رمز باید حداقل هشت کاراکتر و شامل حرف بزرگ، حرف کوچک، عدد و نویسه ویژه باشد.
- تطابق رمز و تکرار آن باید پیش از ارسال روشن باشد.
- قواعد حرف بزرگ و کوچک در متن کاربر صریحاً به حروف انگلیسی اشاره می‌کنند و کنترل نمایش/پنهان‌کردن رمز یک button معنایی با هدف لمسی حداقل ۴۴ پیکسل است.
- پس از شروع ارزیابی رمز، هر قاعده قطعی باید وضعیت متنی/نمادی قبول یا رد داشته باشد؛ neutral فقط پیش از dirtyشدن ورودی مجاز است.
- این checkpoint فقط فرم و security gate را تعیین می‌کند؛ shell نهایی احراز‌شده آن در `0B-2` بسته می‌شود.

## ۳. سناریوهای نماینده

| شناسه | کاربر و نقطه ورود | نتیجه مطلوب |
| --- | --- | --- |
| AUTH-01 | کاربر موجود از `/login` | شماره را وارد، منبع تحویل کد را می‌فهمد و بدون ابهام وارد می‌شود |
| AUTH-02 | کاربر موجود روی دستگاه جدید | انتظار تأیید را می‌فهمد و در نبود دستگاه قبلی وارد recovery می‌شود |
| AUTH-03 | کاربر recovery با درخواست مدرک | فایل و توضیح را ارسال و وضعیت بعدی را از همان context دنبال می‌کند |
| INV-01 | کاربر دعوت‌شده از `/i/:code` | اعتبار، مهلت و مسیرهای واقعاً فعال ثبت‌نام را می‌بیند |
| REG-01 | دعوت‌شده وب | هویت دعوت، OTP پنج‌رقمی، آدرس و اتصال اختیاری تلگرام را مرحله‌به‌مرحله کامل می‌کند |
| REG-02 | کاربر هدایت‌شده از login با `registration_token` | بدون OTP تکراری مستقیماً اطلاعات پایانی را کامل می‌کند |
| SEC-01 | مدیر احراز‌شده | رمز قوی و منطبق را با بازخورد زنده تعریف می‌کند |

## ۴. قاب‌های high-fidelity

مرجع قابل بازبینی در `docs/uiux-stage0b-auth/auth-scenarios.html` ساخته می‌شود:

1. شروع ورود با شماره موبایل؛
2. OTP با تحویل تلگرام و fallback خودکار پیامک؛
3. انتظار تأیید دستگاه و ورود به recovery؛
4. انتظار بررسی مدیریت؛
5. دعوت‌نامه معتبر با دو مسیر فعال؛
6. مرور دعوت در مرحله اول ثبت‌نام؛
7. OTP دقیقاً پنج‌رقمی ثبت‌نام؛
8. تکمیل آدرس در سومین و آخرین مرحله الزامی؛
9. اتصال اختیاری تلگرام پس از تکمیل ثبت‌نام؛
10. ارسال مدرک recovery؛
11. تنظیم رمز عبور مدیر با قواعد زنده؛
12. قاب مستقل `REG-02` برای تکمیل مستقیم پس از login OTP، بدون progress ساختگی یا مسیر بازگشت به OTP ثبت‌نام؛
13. قاب کامل دعوت‌نامه فقط‌وب، recovery غیرمجاز/در‌دسترس‌نبودن بازبین در context تأیید دستگاه، و رمز نامعتبر/نامنطبق؛
14. atlas توسعه‌یافته برای تمام stateهای ساختاری جدول بخش ۷، از جمله terminalهای recovery، خطا/بارگذاری upload، خطا/موفقیت رمز و خطای اتصال اختیاری تلگرام؛
15. چهار نمای کیبورد باز برای موبایل، OTP، آدرس و رمز؛
16. یک نمونه refined و adaptive دسکتاپ recovery با form و context/progress هم‌زمان.

قاب‌ها نماینده قرارداد هستند، نه پیشنهاد حذف stateهای دیگری که در source وجود دارند.

## ۵. قرارداد بصری و تعاملی «مالی مدرن»

- زمینه سرد خنثی، hierarchy سرمه‌ای و CTA آبی؛ طلایی فقط در امضای کوچک برند.
- tokenهای «مالی مدرن» فقط در محدوده auth/public shell تعریف می‌شوند؛ تغییر tokenهای `:root` در این Stage ممنوع است تا primitiveهای مشترک بازار تغییر نکنند.
- تراکم متوسط: اطلاعات مهم نزدیک هم، اما نه فرم دسکتاپ فشرده‌شده در ۳۹۰ پیکسل.
- یک اقدام اصلی در هر مرحله؛ اقدام‌های recovery یا skip از نظر بصری ثانویه‌اند.
- خطا در نزدیک‌ترین context ورودی نمایش داده می‌شود و alert کلی فقط برای خطای فرایند است.
- inputهای موبایل، OTP و رمز در محتوای RTL جهت LTR صحیح دارند.
- اهداف لمسی اصلی حداقل ۴۴ پیکسل و CTA حداقل ۴۸ پیکسل‌اند.
- focus indicator همه کنترل‌های تعاملی از token scoped و solid استفاده می‌کند، حداقل ۳ پیکسل ضخامت دارد و نسبت به هر سطح مجاور آزموده‌شده حداقل `3:1` است؛ state tagهای متن عادی حداقل `4.5:1` هستند.
- glyphهای صرفاً تزئینی notice و picker از assistive technology پنهان می‌شوند؛ اگر معنایی باشند باید نام دسترس‌پذیر عمدی و غیرتکراری داشته باشند.
- فرمت موبایل بدون prefix تکراری است؛ affordance مستقل `09` در کنار مقداری که خودش با `09` شروع می‌شود مجاز نیست.
- marker مشترک footer خنثی است و رنگ سبز فقط برای وضعیت معنایی مثبت استفاده می‌شود.
- متن داخل قاب محصول نتیجه را با فارسی روشن توضیح می‌دهد. واژگان داخلی مانند شناسه opaque، deadline/session storage، API، autocomplete و جزئیات password manager فقط در specification، annotation بازبین یا test باقی می‌مانند.
- حرکت بین مرحله‌ها ۱۴۰ تا ۱۸۰ میلی‌ثانیه، بدون حرکت تزئینی و با پشتیبانی `prefers-reduced-motion` است.
- shell عمومی فقط برای login، invite landing و web registration است و هیچ navigation احراز‌شده یا فضای رزروشده bottom nav ندارد. `/setup-password` احراز‌شده باقی می‌ماند و shell نهایی آن به `0B-2` موکول است.

## ۶. قرارداد موبایل و کیبورد

- content نباید پشت کیبورد یا safe-area پنهان شود.
- CTA مرحله فرم با بازشدن کیبورد قابل‌دسترسی می‌ماند، اما روی فیلد یا متن قرار نمی‌گیرد.
- شماره موبایل `tel` و رمز با password manager سازگار می‌مانند. OTP ورود یک input واحد با `inputmode=numeric` و `autocomplete=one-time-code` است؛ OTP ثبت‌نام نیز یک input واحد و دقیقاً پنج‌رقمی است، اما semantics آن جداگانه تنظیم می‌شود و نبود autocomplete در runtime فعلی قرارداد الزام‌آور طراحی نیست.
- در ورود، Back از OTP به شماره برمی‌گردد. در `REG-01` Back فقط در همان مسیر سه‌مرحله‌ای عمل می‌کند و در `REG-02` هرگز کاربر را وارد OTP تکراری ثبت‌نام نمی‌کند.
- حفظ state ثبت‌نام هنگام Back، refresh و خطای موقت باید در پیاده‌سازی و آزمون پذیرش اثبات شود؛ این گزاره ادعای اثبات رفتار فعلی `WebRegister` نیست.
- افشای آدرس باید هنگام آزمون کیبورد واقعی همچنان دیده شود یا بدون ازدست‌دادن context فیلد به‌آسانی قابل‌دسترسی باشد؛ illustration ثابت صرفاً intent را نشان می‌دهد.
- تغییر visibility یا refresh نباید deadline نمایشی را از زمان authoritative جدا کند.

## ۷. state matrix الزامی

| خانواده | stateهای قابل تشخیص |
| --- | --- |
| دعوت | lookup loading، pending معتبر، فقط وب، هر دو مسیر، completed redirect، expired/invalid |
| ورود | شماره خالی/نامعتبر، request loading، OTP فعال بازیابی‌شده، کد اشتباه، fallback خودکار، resend legacy، network/PWA recovery |
| دستگاه | waiting approval، approved، rejected، expired |
| recovery | pending admin، identity requested، upload invalid/loading، submitted، approved، rejected، expired، cancelled |
| ثبت‌نام | invite loading/error، step 1، OTP پنج‌رقمی، آدرس نامعتبر، submit loading/error، Telegram optional، skip/success |
| رمز | rule incomplete، mismatch، submit loading، API error، success |

## ۸. اثبات adaptive دسکتاپ

در دسکتاپ فرم بی‌دلیل پهن نمی‌شود. سناریوی پرتراکم ارسال مدرک recovery در ستون با عرض محدود باقی می‌ماند و فضای اضافه برای context همان درخواست، progress، deadline و راهنمای امنیتی استفاده می‌شود. همین یک proof refined برای Stage 0B-1 کافی است و سناریوی دوم دسکتاپ الزامی نیست. پیاده‌سازی می‌تواند variant چیدمان scoped در خانواده `ui-*` اضافه کند؛ ایجاد خانواده سراسری سوم یا تغییر tokenهای `:root` مجاز نیست. این چیدمان مجوز افزودن اطلاعات یا رفتار جدید نیست.

## ۹. non-goalها

- تغییر endpoint، payload، نقش، permission، timer یا polling؛
- ادغام یا حذف مرحله‌های ثبت‌نام؛
- تغییر سیاست OTP، session approval یا identity review؛
- بررسی یا تغییر بازار و پیام‌رسان؛
- شروع کدنویسی محصول یا انتشار.
- قطعی‌کردن shell نهایی مسیر احراز‌شده `/setup-password` پیش از تصمیم Stage 0B-2؛ در این checkpoint فقط خود فرم و security gate سنجیده می‌شود.

## ۱۰. معیار پذیرش checkpoint

- مسیرهای AUTH-01 تا SEC-01 در نمونه‌ها و state atlas قابل ردیابی باشند.
- fit عمودی و افقی با روشی که ارتفاع طبیعی محتوا و stateهای خطا را می‌سنجد، در عرض‌های هدف ثبت شود؛ ادعا فقط به ابعاد و elementهای واقعاً اندازه‌گیری‌شده محدود می‌ماند.
- تشخیص مرحله، اقدام اصلی و راه خروج/بازیابی در آزمون پنج‌ثانیه‌ای ممکن باشد.
- دعوت، ورود، ثبت‌نام و recovery شبیه یک خانواده واحد باشند اما از نظر عنوان و progress با هم اشتباه نشوند.
- ثبت‌نام سه مرحله الزامی را نشان دهد و اتصال تلگرام فقط به‌عنوان مرحله بعدی اختیاری معرفی شود.
- خطا، انتظار و موفقیت فقط با رنگ منتقل نشوند.
- نمونه دسکتاپ adaptive باشد، نه نسخه کشیده موبایل.
- بازار، پیام‌رسان و کد محصول تغییر نکرده باشند.
- آمار touch target همه کنترل‌های معنایی، از جمله password visibility، را پوشش دهد یا exclusionها را صریح اعلام کند؛ عدد مربوط به elementهای دستی علامت‌گذاری‌شده نباید به همه کنترل‌ها تعمیم داده شود.
- آستانه ۴۸ پیکسل CTA جدا از آستانه عمومی ۴۴ پیکسل سنجیده و fail-closed شود؛ variant ثانویه/خطر/ghost نیز نباید صرفاً به‌دلیل غیر-primaryبودن از سنجش CTA خارج شود.
- Tab traversal مصنوعی، رنگ/ضخامت focus، glyph source audit و contrast tokenها فقط artifact ثابت و جفت‌های scoped را می‌سنجند؛ screen reader، accessibility tree و runtime WebView باید در implementation gate آزموده شوند.
- تصویر static کیبورد، OTP بخش‌بندی‌شده، font-load محلی capture و JUnit فعلی به‌ترتیب فقط design intent، presentation، فونت همین محیط capture و رفتار موجود را ثابت می‌کنند؛ این شواهد جای آزمون runtime کیبورد، input معنایی، تحویل production font یا پیاده‌سازی آینده را نمی‌گیرند.
- natural-fit clone فقط قاب‌های authored در normal flow و عرض‌های ثبت‌شده را پوشش می‌دهد؛ absolute/fixed descendant، zoom، font scaling، متن پویای بلندتر و containing block متفاوت نیازمند assertion پیاده‌سازی جداگانه‌اند.

## ۱۱. گیت تصمیم مالک محصول

تصمیم‌های پیشنهادی زیر از disposition شواهد به دست آمده‌اند و برای بستن checkpoint به تأیید مالک محصول نیاز دارند:

- shell عمومی دقیقاً به login، invite landing و web registration محدود است؛ `/setup-password` احراز‌شده باقی می‌ماند و shell نهایی آن در `0B-2` تصمیم‌گیری می‌شود؛
- `REG-01` سه مرحله الزامی دارد و اتصال تلگرام یک پیشنهاد اختیاریِ مستقل پس از تکمیل است؛ `REG-02` تکمیل مستقیم و بدون «۳ از ۳» ساختگی یا Back به OTP تکراری است؛
- Modern Finance در auth/public shell scoped است، `:root` تغییر نمی‌کند و public shell هیچ bottom-nav reservation ندارد؛
- متن polling، آدرس و مدرک هویتی صادق، محدود و مستقل از جزئیات ناپایدار پیاده‌سازی است؛
- قاب‌های ساختاری مشخص‌شده و atlas کامل می‌شوند، و یک proof refined دسکتاپ کافی است؛
- ادعای evidence فقط به آنچه واقعاً اندازه‌گیری یا آزموده شده محدود می‌شود؛
- persistence و Back/refresh ثبت‌نام، semantics واقعی inputها، keyboard و shell integration معیار پذیرش پیاده‌سازی‌اند و نه ادعای runtime فعلی.
- کپی محصول، افشای نزدیک فیلد/اقدام، feedback کامل قاعده‌های رمز، focus solid، CTA حداقل ۴۸، نام دسترس‌پذیر بدون glyph تزئینی، ورودی موبایل بدون prefix تکراری و footer خنثی قرارداد طراحی مصوب این checkpoint هستند.

تأیید این تصمیم‌ها فقط مرز طراحی لازم برای بستن Stage 0B-1 را می‌بندد. ورود به `0B-2 — خانه و پوسته احراز‌شده` علاوه بر تأیید مالک، به review gate چهار‌نقشی روی آخرین commit نیاز دارد؛ هنوز مجوز کدنویسی محصول، تغییر endpoint، تغییر بازار/پیام‌رسان، merge یا انتشار نیست.
