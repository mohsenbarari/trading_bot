# سند حل چالش‌های فیچر مشتری - نسخه چهارم

این فایل در وضعیت فعلی roadmap اجرایی نیست. این سند باید قبل از هر فاز پیاده‌سازی، تصمیم‌های محصولی، قواعد دسترسی، حریم خصوصی، منطق کمیسیون، محدودیت‌های پیام‌رسان/گروه/کانال، و lifecycle فیچر مشتری را ببندد.

این نسخه چهارم بر اساس پاسخ‌های تکمیلی کاربر ساخته شده است. هنوز وارد تحلیل ساختار کد پروژه نشده‌ایم و تا وقتی challengeهای باز این سند بسته نشده‌اند، ورود به تحلیل فنی و ساخت roadmap اجرایی دقیق مجاز نیست.

راهنمای وضعیت:
- `[x]` تصمیم نهایی‌شده یا requirement قطعی از سمت محصول
- `[ ]` ابهام باز، decision blocker، یا موردی که باید قبل از ورود به فاز فنی بسته شود

## 0. قاعده استفاده از این سند

- [x] این فایل در این مرحله «سند بستن چالش‌ها» است، نه «نقشه راه اجرا».
- [x] این نسخه بر اساس توضیحات محصولی کاربر نوشته شده و عمداً وارد ساختار کد پروژه نشده است.
- [x] تا وقتی سوال‌های باز این سند بسته نشده‌اند، تحلیل فنی مبتنی بر کد و phase‌بندی اجرا نباید شروع شود.
- [x] فقط بعد از بسته‌شدن challengeهای باز این سند، می‌توان وارد بررسی کد پروژه و طرح سوال‌های فنی شد.

## 1. هدف و دامنه

- [x] «مشتری» یک هویت واقعی در سیستم است، نه صرفاً یک برچسب نمایشی در فرانت.
- [x] هر مشتری دقیقاً زیرمجموعه‌ی یک owner است و این فاز چندمالکیتی را پشتیبانی نمی‌کند.
- [x] مشتری فقط از طریق وب‌اپ از سیستم استفاده می‌کند.
- [x] مشتری به بازار و پیام‌رسان دسترسی دارد، اما هر دو دسترسی محدود و وابسته به owner هستند.
- [x] این فیچر باید هم‌زمان owner، حسابداران owner، ادمین‌ها، و سایر کاربران عادی را پوشش دهد.
- [x] این سند فعلاً فقط behavior محصولی را می‌بندد و وارد مدل داده، routeها، migrationها، یا ساختار فعلی پروژه نمی‌شود.

## 2. نقش‌ها و اصطلاحات

- [x] `owner`: کاربر دعوت‌کننده و مدیر مشتری.
- [x] `customer`: کاربر وابسته به یک owner با دسترسی محدود به بازار و پیام‌رسان.
- [x] `accountant`: حسابدار owner که در برخی سطوح همان tree دسترسی owner را نسبت به مشتری دارد.
- [x] `admin`: نقش مدیریتی که باید سقف تعداد مشتری هر owner را کم یا زیاد کند و هر چیزی را که owner می‌بیند ببیند، به‌جز بخش مدیریت مشتریان.
- [x] `management name`: نام یکتای مشتری در محدوده‌ی همان owner که برای مدیریت بهتر در بازار و پیام‌رسان استفاده می‌شود.

## 2.1. سطح‌بندی مشتری

- [x] مدل customer دو سطح دارد: `Tier 1` و `Tier 2`.
- [x] `Tier 1`: مشتری حق ثبت آفر دارد و بازار را مانند کاربر عادی می‌بیند؛ کمیسیون این سطح خارج از پلتفرم و توافقی بین owner و customer است.
- [x] `Tier 2`: مشتری حق ثبت آفر ندارد و فقط market consume/execute می‌کند؛ کمیسیون این سطح داخل پلتفرم و طبق policy فعلی owner اعمال می‌شود.
- [x] تفسیر و enforcement باید tier-aware باشد و ruleهای هر سطح در backend به‌صورت authoritative اعمال شوند.

## 3. قواعد هویت، مالکیت، و ظرفیت

- [x] هر کاربر عادی به‌صورت پیش‌فرض می‌تواند حداکثر `5` مشتری داشته باشد.
- [x] ادمین می‌تواند این سقف را برای هر owner به‌صورت مجزا کم یا زیاد کند.
- [x] هر مشتری فقط می‌تواند متعلق به یک owner باشد.
- [x] مشتری کاربر1 نمی‌تواند هم‌زمان مشتری کاربر2 هم باشد.
- [x] ورود یا احراز هویت مشتری به `username` وابسته نیست.
- [x] مشتریان هر owner باید یک `management name` یکتا در محدوده‌ی همان owner داشته باشند.
- [x] این نام یکتا برای مدیریت بهتر مشتری‌ها در بازار و پیام‌رسان لازم است، نه برای login/auth.
- [x] `management name` مشتری بعد از ایجاد باید immutable بماند.
- [x] owner باید بتواند از پروفایل خودش مشتری اضافه کند.
- [x] owner باید بتواند از پروفایل خودش لیست مشتریان خودش را ببیند.
- [x] owner باید بتواند از همان بخش مشتری حذف کند.
- [x] کاربران عادی دیگر نباید لیست مشتریان ownerهای دیگر را در پروفایل عمومی آن‌ها ببینند.
- [x] ادمین‌ها باید بتوانند لیست مشتریان را ببینند.

## 4. پروفایل عمومی و حریم خصوصی

- [x] مشتری پروفایل عمومی دارد.
- [x] پروفایل عمومی مشتری فقط باید برای owner او، حسابداران owner او، و ادمین‌ها قابل مشاهده باشد.
- [x] مشتری فقط باید بتواند پروفایل عمومی owner خودش را ببیند.
- [x] حسابداران پروفایل مستقل ندارند.
- [x] هر لینک پروفایل حسابدار باید به پروفایل owner اصلی resolve شود.
- [x] ادمین باید هر چیزی را که owner در سطوح نمایشی می‌بیند ببیند، اما بخش مدیریت مشتریان owner-only باقی می‌ماند.
- [x] سایر کاربران عادی نباید بتوانند پروفایل عمومی مشتری را ببینند.
- [x] لیست مشتریان owner نباید در پروفایل عمومی owner برای کاربران عادی دیگر نمایش داده شود.
- [x] ادمین در سطح نمایش/history/profile باید همان visibility owner را نسبت به مشتری داشته باشد.

## 5. پیام‌رسان، گروه، و کانال

- [x] مشتری به پیام‌رسان دسترسی دارد.
- [x] مشتری می‌تواند در دایرکت به owner و حسابداران owner پیام بدهد.
- [x] مشتری فقط باید بتواند با owner و حسابداران owner خودش در پیام‌رسان در ارتباط باشد.
- [x] مشتریان به هیچ وجه نباید مستقیماً با یکدیگر در ارتباط باشند.
- [x] مشتری نباید بتواند با کاربران خارج از tree مربوط به owner خودش در ارتباط مستقیم باشد.
- [x] مشتری در هیچ کانالی نباید عضو شود.
- [x] مشتری نباید گروه بسازد.
- [x] مشتری فقط در گروه‌هایی می‌تواند عضو شود که اعضای آن محدود به owner، حسابداران owner، و همان یک مشتری باشند.
- [x] هیچ گروهی که در آن بیش از یک مشتری حضور داشته باشد مجاز نیست.

## 6. مدیریت مشتری توسط owner

- [x] owner باید در پروفایل خودش یک بخش مجزا برای مدیریت مشتریان داشته باشد.
- [x] این بخش باید حداقل شامل لیست مشتریان، افزودن مشتری، و حذف مشتری باشد.
- [x] فقط owner حق مدیریت مشتریان خودش را دارد.
- [x] حسابداران owner فقط مجاز به دیدن مشتریان و تاریخچه معاملات آن‌ها هستند و حق مدیریت مشتری را ندارند.
- [x] owner باید بتواند برای مشتریان `Tier 2` نرخ کمیسیون مشخص کند.
- [x] نرخ کمیسیون برای هر مشتری `Tier 2` owner می‌تواند متفاوت باشد.
- [x] مدیریت و تغییر نرخ کمیسیون فقط بر عهده‌ی owner است.
- [x] برای `Tier 1` هیچ نرخ کمیسیون سیستمی در تنظیمات مشتری نگهداری نمی‌شود؛ توافق کمیسیون خارج از پلتفرم است.
- [x] مشتری نباید از نرخ کمیسیون خودش مطلع شود.
- [x] owner باید بتواند برای هر مشتری به‌صورت مجزا حداقل تعداد کالا در هر معامله را تعیین کند.
- [x] owner باید بتواند برای هر مشتری به‌صورت مجزا حداکثر تعداد کالا در هر معامله را تعیین کند.
- [x] حداقل و حداکثر تعداد کالا برای هر مشتری نباید از بازه‌ی حداقل/حداکثر تعیین‌شده توسط ادمین کمتر یا بیشتر شود.
- [x] owner باید بتواند برای هر مشتری تعداد معاملات مجاز در روز تعیین کند.
- [x] owner باید بتواند برای هر مشتری سقف تعداد کل کالای معامله‌شده در روز تعیین کند.
- [x] owner باید بتواند مشتری را برای مدت زمان مشخص از معامله منع کند.

## 7. منطق کمیسیون (Tier 2)

- [x] منطق کمیسیون سیستمی فقط برای `Tier 2` اعمال می‌شود.
- [x] `Tier 1` به‌صورت سیستمی commission-adjusted price ندارد.

- [x] ترجیح محصول در نسخه فعلی این است که کمیسیون درصدی باشد، نه عدد ثابت به ازای هر کالا.
- [x] نرخ کمیسیون باید حداکثر تا یک رقم اعشار را پشتیبانی کند.
- [x] گام تغییر نرخ کمیسیون در UI باید `0.1` باشد.
- [x] در بخش مدیریت مشتریان، بالای کادر کمیسیون باید یک پیام کوچک کمکی نمایش داده شود.
- [x] این پیام کمکی باید با هر تغییر درصد، کمیسیون متناظر به ازای هر `100,000,000` را محاسبه و نمایش دهد.
- [x] مثال قطعی محصول: اگر نرخ کمیسیون `1%` باشد، متن کمکی باید معادل «کمیسیون شما به ازای هر 100 میلیون 1,000,000 می‌شود» را نشان دهد.
- [x] این متن کمکی باید به‌صورت زنده با هر تغییر درصد آپدیت شود.
- [x] مشتری نباید خود نرخ کمیسیون را ببیند.
- [x] روش authoritative رند کردن برای `Tier 2` این است که در آفرهای خرید همیشه `floor_100(raw_price - commission)` و در آفرهای فروش همیشه `ceil_100(raw_price + commission)` اعمال شود.
- [x] عدد رندشده مبنای اصلی قیمت در منطق محصولی مشتری‌محور است.
- [x] در این مدل، عدد معامله‌ی بین owner و مشتری تغییر نمی‌کند و اختلاف ناشی از رندینگ فقط داخل مؤلفه‌ی کمیسیون پنهان می‌ماند.
- [x] nearest-100 rule برای customer استفاده نمی‌شود و ambiguity مربوط به midpoint با همین floor/ceil direction-fixed بسته شده است.

## 8. منطق بازار و نمایش آفرها

### 8.1. بازار برای Tier 1

- [x] `Tier 1` بازار را مانند کاربران عادی می‌بیند.
- [x] آفر ثبت‌شده توسط `Tier 1` با همان raw price برای همه کاربران منتشر می‌شود.
- [x] تنها تفاوت نمایش: owner همان مشتری روی آفر مشتری خودش تگ «مشتری + نام مشتری» می‌بیند.
- [x] middle/super admin در market نسبت به آفر Tier 1 visibility ویژه ندارند و مانند سایر viewerها همان raw published price را می‌بینند.

### 8.2. بازار برای Tier 2

- [x] `Tier 2` اجازه ثبت آفر ندارد.
- [x] `Tier 2` فقط می‌تواند آفرهای موجود را ببیند و روی آن‌ها درخواست معامله بدهد.
- [x] اگر نرخ کمیسیون `Tier 2` برای آن customer تنظیم نشده باشد یا صفر باشد، market را با همان raw published price می‌بیند.
- [x] اگر `Tier 2` یک آفر خرید را ببیند، باید کمیسیون viewer-specific از raw price کسر شود.
- [x] اگر `Tier 2` یک آفر فروش را ببیند، باید کمیسیون viewer-specific به raw price افزوده شود.
- [x] در projection آفرهای خرید برای `Tier 2`، عدد بعد از کسر کمیسیون به سمت پایینِ مضرب `100` گرد می‌شود.
- [x] در projection آفرهای فروش برای `Tier 2`، عدد بعد از افزودن کمیسیون به سمت بالای مضرب `100` گرد می‌شود.
- [x] مثال قطعی: آفر خرید raw=`192800` برای `Tier 2` با کمیسیون `0.5%` به `191800` و با کمیسیون `0.7%` به `191400` نمایش داده می‌شود.
- [x] مثال قطعی: آفر فروش raw=`53500` برای `Tier 2` با کمیسیون `0.5%` به `53800` و با کمیسیون `0.7%` به `53900` نمایش داده می‌شود.
- [x] fair-price / competitive-price برای آفرهای `Tier 1` دقیقاً باید مثل سایر کاربران و بر اساس raw price همان آفر محاسبه شود.
- [x] `Tier 2` چون offer source نیست، branch جداگانه‌ای در fair-price calculation ایجاد نمی‌کند و viewer projection او وارد dataset محاسبه‌ی قیمت منصفانه نمی‌شود.

- [x] `Tier 2` market جداگانه‌ای ندارد و فقط price projection آفرها برای او با policy خودش تغییر می‌کند.
- [x] owner فقط روی آفرهای `Tier 1` مشتریان خودش تگ کوچک «مشتری» را در market می‌بیند، چون `Tier 2` خودِ آفر منتشر نمی‌کند.
- [x] در بازار، نقش `MIDDLE_MANAGER` و `SUPER_ADMIN` به‌خودی‌خود نباید raw visibility ویژه ایجاد کند؛ ادمین فقط وقتی می‌تواند raw price آفر مشتری را ببیند که خودش owner همان مشتری باشد.
- [x] حسابداران به بازار دسترسی ندارند.
- [x] برای هر viewer غیر-owner همان customer، آفر `Tier 1` customer نباید از نظر هویت یا ظاهر هیچ تفاوتی با سایر آفرها داشته باشد.
- [x] در صفحه بازار هیچ هویت مستقیمی از ثبت‌کننده‌ی آفر نباید نمایش داده شود.
- [x] تنها تگ مجازِ customer-aware در بازار، تگ «مشتری» روی آفرهای `Tier 1` مشتریان همان owner برای خود owner آن relation است؛ این visibility از رابطه می‌آید نه از role ادمینی.

## 9. منطق معامله و history

### 9.0. ماتریس نهایی Brokerage / Chained Settlement

- [x] این ماتریس پوشش کامل دسته‌بندی actorها است: offer source فقط می‌تواند `Owner` یا `Tier 1` باشد، چون `Tier 2` حق ثبت آفر ندارد.
- [x] اگر buy/sell polarity یک حالت برعکس شود، chain actorها عوض نمی‌شود؛ فقط نقش‌های buyer/seller در همان legها بر اساس offer type معکوس می‌شوند و قانون price مربوط به همان offer type اعمال می‌شود.
- [x] اگر actor عادی customer نداشته باشد، از نظر این ماتریس در دسته‌ی `Owner`/principal مستقیم قرار می‌گیرد.
- [x] برای نمونه‌های زیر گروه A شامل `رامین` به‌عنوان owner، `سینا` و `سهراب` به‌عنوان `Tier 1`، و `علی` به‌عنوان `Tier 2` است؛ گروه B شامل `مجید` به‌عنوان owner، `پیمان` به‌عنوان `Tier 1`، و `محمد` به‌عنوان `Tier 2` است.

- [x] حالت 11: source=`Tier 1` و responder=`Tier 2` owner دیگر. اگر raw buy=`100000` و commission=`0.7%` باشد، نمونه: `سینا ←buyer / رامین ←seller @ 100000`، `رامین ←buyer / مجید ←seller @ 100000`، و `مجید ←buyer / محمد ←seller @ 99300`.

### 9.1. قرارداد قطعی اجرای معامله برای Tier 1 (سناریوهای تاییدشده)

- [x] حالت 1 (Tier1 seller vs Tier1 buyer از owner دیگر): chain سه‌مرحله‌ای با قیمت یکسان ثبت می‌شود.
- [x] حالت 2 (Tier1 buyer vs owner دیگر): chain دو‌مرحله‌ای با قیمت یکسان ثبت می‌شود.
- [x] حالت 3 (owner offer vs Tier1 responder): chain دو‌مرحله‌ای با قیمت یکسان ثبت می‌شود.
- [x] حالت 4 (owner offer vs Tier1 own-customer): trade مستقیم owner ↔ customer با قیمت یکسان ثبت می‌شود.
- [x] در تمام حالت‌های Tier 1، نرخ کمیسیون داخل پلتفرم وارد pricing pipeline نمی‌شود.
- [x] مثال قطعی حالت 1: `سینا` sell raw=`52000` publish می‌کند، `پیمان` request می‌زند، و chain نهایی این است: `پیمان ←buyer / مجید ←seller @ 52000`، `مجید ←buyer / رامین ←seller @ 52000`، `رامین ←buyer / سینا ←seller @ 52000`.
- [x] مثال قطعی حالت 2: `پیمان` buy raw=`98000` publish می‌کند، `رامین` request می‌زند، و chain نهایی این است: `مجید ←buyer / رامین ←seller @ 98000`، `پیمان ←buyer / مجید ←seller @ 98000`.
- [x] مثال قطعی حالت 3: `مجید` buy raw=`189600` publish می‌کند، `سینا` request می‌زند، و chain نهایی این است: `رامین ←buyer / سینا ←seller @ 189600`، `مجید ←buyer / رامین ←seller @ 189600`.
- [x] مثال قطعی حالت 4: `رامین` sell raw=`188000` publish می‌کند، `سینا` request می‌زند، و فقط یک trade مستقیم `سینا ←buyer / رامین ←seller @ 188000` ثبت می‌شود.

- [x] اگر مشتری1 از مشتریان owner1 از user3 کالایی بخرد، از دید business relation این اتفاق به‌صورت «user3 به owner1 فروخته و owner1 به مشتری1 فروخته» تفسیر می‌شود.
- [x] این relation باید در history و نمایش context معامله منعکس شود.
- [x] مشتری‌ها نباید مستقیماً با طرف‌های دیگر معامله ثبت‌شده‌ی نهایی داشته باشند؛ معامله باید حتماً از ownerِ همان customer عبور کند و زنجیره‌ی trade rows بر همین اساس ساخته شود.
- [x] اگر هر دو سمت معامله customer باشند و ownerهای متفاوت داشته باشند، یک معامله business واحد باید به‌صورت زنجیره‌ای در چند trade row ثبت شود، نه یک trade مستقیم customer ↔ customer.
- [x] نوتیفیکیشن‌ها، history طرفین، و projectionهای UI باید با همین owner-mediated chain هم‌راستا باشند، نه با یک trade مستقیم customer ↔ customer.
- [x] chain اجرای معامله برای `Tier 2` در سناریوهای owner-offer و outsider-offer از سمت responder قطعی شده است.
- [x] اگر `Tier 2` روی آفر owner خودش request بزند، trade مستقیم owner ↔ customer با همان قیمت projected-to-customer ثبت می‌شود.
- [x] مثال قطعی: owner=`شهاب` buy raw=`192800`، customer=`محمد` با commission `0.7%` request می‌زند، trade مستقیم `شهاب ←buyer / محمد ←seller @ 191400` ثبت می‌شود.
- [x] اگر `Tier 2` روی آفر owner یا actor مجازِ بیرون از owner خودش request بزند، execution بر اساس source category به chain دو یا سه‌مرحله‌ای می‌شکند.
- [x] در Tier 2، responder همیشه از owner خودش عبور می‌کند؛ اگر source هم customer/`Tier 1` باشد، source side نیز از owner خودش عبور می‌کند.
- [x] legهای raw همیشه price خام offer را حفظ می‌کنند و leg customer `Tier 2` ↔ owner با `viewer_effective_price` همان customer ثبت می‌شود.
- [x] در buy-offer بیرونی با source owner/non-customer: customer responder با قیمت viewer-specific به owner خودش trade می‌کند و سپس owner او با raw price به buyer/source actor اصلی trade می‌کند.
- [x] مثال قطعی: owner=`شهاب` buy raw=`192800`، customer=`علی` با commission `0.5%` request می‌زند، trade1 = `متین ←buyer / علی ←seller @ 191800` و trade2 = `شهاب ←buyer / متین ←seller @ 192800`.
- [x] در sell-offer بیرونی با source owner/non-customer: customer responder با قیمت viewer-specific از owner خودش buy می‌کند و سپس owner او با raw price از seller/source actor اصلی buy می‌کند.
- [x] اگر source آفر یک `Tier 1` باشد، source-owner نیز در chain حفظ می‌شود؛ نمونه‌های قطعی این حالت در ماتریس 9.0 حالت‌های 10 و 11 ثبت شده‌اند.
- [x] برای sell-offer بیرونی از source=`Tier 1` و responder=`Tier 2` owner دیگر، mirror حالت 11 اعمال می‌شود: responder customer با `viewer_effective_price` از owner خودش می‌خرد، responder owner با raw price از source owner می‌خرد، و source owner با raw price از source `Tier 1` می‌خرد.
- [x] اگر owner1 تاریخچه معاملات خودش با user3 را از پروفایل عمومی user3 ببیند، روی معامله‌هایی که مربوط به مشتری1 هستند باید یک تگ کوچک «مشتری» و در کنار آن نام مشتری1 را ببیند.
- [x] اگر user3 تاریخچه معاملات با owner1 را ببیند، برای همان معامله نباید آن تگ مشتری را ببیند.
- [x] اگر owner1 در حال مرور تاریخچه معاملات با مشتری1 باشد، در هر معامله باید نام سمت دیگر معامله همراه با یک تگ ریز context نمایش داده شود.
- [x] حسابداران owner فقط مجاز به دیدن مشتریان و تاریخچه معاملات مشتریان هستند.

## 12. سوال‌ها و ابهام‌های باز برای نسخه‌چهارم یا ورود به فاز فنی

- [x] مرجع authoritative برای matching, ranking, validation, و execution در کدهای فعلی باید بر این اصل بسته شود که مشتری یک کاربر عادی اما محدود است؛ هر رفتار استاندارد کاربر عادی برای مشتری مجاز است مگر آن‌جا که صراحتاً یکی از محدودیت‌های مشتری آن رفتار را منع کرده باشد. در لایه business identity نیز owner مرجع رابطه و سیاست‌ها است، اما actor واقعی می‌تواند خود مشتری باشد.
- [x] lifecycle حذف مشتری نیز باید با همین اصل بسته شود: تا وقتی محدودیت‌های تعریف‌شده نقض نشوند، با مشتری مانند کاربر عادی رفتار می‌شود. بنابراین حذف/غیرفعال‌سازی/قطع رابطه در تحلیل فنی باید از مسیرهای موجود کاربر عادی استفاده کند و فقط لایه‌ی relation owner ← customer و محدودیت‌های اختصاصی مشتری را به آن اضافه کند، نه یک lifecycle کاملاً جدا و استثنایی.
